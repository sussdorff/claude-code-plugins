## Architecture Principles

> *Why the codebase is structured the way it is — not what, but why.*

1. **Marketplace-as-monorepo**: `.claude-plugin/marketplace.json` catalogs 8 plugins, each a
   self-contained subdirectory. A single install target gives users access to every plugin
   without cross-repo dependencies.
2. **Plugins are category bundles, not individual tools**: Each top-level directory (`infra/`,
   `business/`, `content/`, `medical/`, `dev-tools/`, `meta/`, `core/`, `beads-workflow/`) is
   one plugin grouping related skills, agents, and hooks. Users install by theme, not by tool.
3. **Skill-first, agent-second, hook-last**: Prefer inline skills for lightweight tasks; reach
   for subagents only for multi-step autonomous work that benefits from context isolation; use
   hooks only for deterministic lifecycle automation (session-start, pre-compact, cache
   invalidation). Visible in the agent/hook counts per plugin — skills dominate.
4. **Self-contained Python scripts (PEP 723)**: Scripts with external deps use inline metadata
   (`#!/usr/bin/env -S uv run --quiet --script`). Scripts get copied into `~/.claude/` during
   install — separate `requirements.txt`/`pyproject.toml` would not travel with them.
   Stdlib-only scripts may use plain `python3`.
5. **Symlinked `.claude/` for development**: This repo's own `.claude/` is a symlink into
   `~/code/claude/claude-code-plugins/` so harness state lives outside the marketplace it
   ships. Prevents accidental publishing of local session state.
6. **Beads for all task tracking**: Work items, refactor notes, session learnings, and
   dependencies are captured in `bd` — never in `TodoWrite`, `TaskCreate`, or markdown TODOs.
   This keeps task state durable across sessions and syncable via Dolt.
7. **Patterns extracted, not frameworks imported**: Architectural docs (`docs/architecture/`)
   explicitly analyze BMAD / Superpowers / Spec Kit but extract individual patterns into
   dedicated skills rather than importing whole frameworks. Beads remains the single source
   of truth.

## Established Patterns

> *Recurring implementation patterns the team has settled on.*

### Skill YAML Frontmatter
**Where**: `*/skills/*/SKILL.md`
**Pattern**: Every skill starts with `---` YAML frontmatter containing `name`, `description`,
and optionally `argument-hint`. The description carries the trigger phrases Claude uses for
auto-discovery.
**Why**: Claude Code requires frontmatter for skill registration; trigger phrases in
`description` drive the skill-matching heuristic.

### Plugin Manifest Minimalism
**Where**: `*/.claude-plugin/plugin.json`
**Pattern**: Each plugin.json contains only `name`, `version`, `description`, `keywords`.
Never `category` (that field belongs in `marketplace.json` only — it triggers validation
errors inside `plugin.json`).
**Why**: Claude Code's plugin validator rejects unknown fields. Categorization is a
marketplace-level concern, not a plugin-level one.

### PEP 723 Inline Scripts
**Where**: `*/skills/*/scripts/*.py`, `*/hooks/*.py`
**Pattern**: Python scripts declare dependencies in a `# /// script` block and shebang
`#!/usr/bin/env -S uv run --quiet --script`. Stdlib-only scripts fall back to plain
`#!/usr/bin/env python3`.
**Why**: Scripts are copied into `~/.claude/` during install and must run standalone without
a surrounding virtualenv. PEP 723 lets each script carry its own dependency spec.

### Agent-as-Markdown
**Where**: `*/agents/*.md`
**Pattern**: Each subagent is a single markdown file with YAML frontmatter (`name`,
`description`, `tools`) followed by the system prompt body. No code — Claude Code materializes
the agent from the markdown at runtime.
**Why**: Keeps agents reviewable, diff-friendly, and composable. No build step.

### Lifecycle Hooks in Python
**Where**: `beads-workflow/hooks/*.py`, `core/hooks/*.py`, `dev-tools/hooks/*.py`
**Pattern**: Hooks are small Python scripts registered in the plugin's `hooks/hooks.json`.
They follow the three-tier exit code convention (0=info/pass, 1=hook-error, 2=block).
**Why**: Deterministic side-effects (cache invalidation, session priming, state capture)
belong in hooks — not in the model's reasoning path.

### Session-Close Double-Merge
**Where**: `core/agents/session-close.md`
**Pattern**: Session end always (1) merges `main` into the feature branch up front, (2)
rebases/merges again before pushing, (3) updates changelog + CalVer tag, (4) persists
learnings to open-brain, (5) pushes code then closes the bead.
**Why**: Parallel bead work across worktrees creates race conditions on `main`. The
double-merge ensures the feature branch is current at both checkpoints.

### Architecture-as-Dated-Folder
**Where**: `docs/architecture/YYYY-MM-DD-<topic>/`
**Pattern**: Design sessions produce numbered markdown files (`00-README.md`,
`01-*.md`, ...) grouped under a date-prefixed folder. The `00-README.md` lists decisions
and cross-references.
**Why**: Captures the reasoning trail without overwriting earlier context. Future Claude
sessions can read decisions + their justifications without interviewing the author.

## Critical Invariants

> *Rules that MUST hold across the entire codebase.*

1. **No `category` field in `plugin.json`** — it belongs in `marketplace.json` only.
   Including it breaks plugin validation.
2. **Every skill's `SKILL.md` has YAML frontmatter with `name` and `description`** — otherwise
   the skill is invisible to Claude Code's skill-matching.
3. **Python scripts with external deps MUST use PEP 723 inline metadata** — separate
   `requirements.txt`/`pyproject.toml` files do not travel with installed skills.
4. **Source code (including comments, identifiers, log messages, string literals) is English**
   — user-facing strings may be localized when the project requires it, but source artifacts
   are not.
5. **Never use `TodoWrite`, `TaskCreate`, or markdown TODOs for task tracking** — use `bd` only.
   Task state lives in beads so it survives across sessions and syncs via Dolt.
6. **Raw `dolt push --force` is BLOCKED** — use the beads wrapper `bd dolt push --force`
   (required to work around upstream Dolt bug dolthub/dolt#10807). `bd init --force` on
   existing projects is also blocked — it drops databases.
7. **Work is not done until `git push` succeeds** — session close runs `git pull --rebase`
   → `bd dolt push` → `git push` → `git status` verification. Stopping before push strands
   work locally.
8. **`.claude/` directories are git-ignored** — the project's own harness state is a symlink
   into `~/code/claude/` and must not be committed into the marketplace it ships.
9. **Hooks follow the three-tier exit code convention** — `0` for info/pass, `1` for hook
   error (surfaces to user, does not block), `2` for block. Deviating breaks Claude Code's
   hook contract.
